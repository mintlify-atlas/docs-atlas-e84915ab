## Overview

API keys are used to authenticate requests to the Cencori API. Each key belongs to a project and can be scoped by environment.

## Authentication

API key management endpoints require dashboard authentication (user session).

## List API Keys

```
GET /api/projects/{projectId}/api-keys
```

List all API keys for a project.

### Query Parameters

<ParamField query="environment" type="string">
  Filter by environment: `production` or `test`
  
  Default: `production`
</ParamField>

### Response

```json
{
  "apiKeys": [
    {
      "id": "key_abc123",
      "name": "Production API Key",
      "key_prefix": "csk_abc1...",
      "environment": "production",
      "key_type": "secret",
      "created_at": "2024-01-01T00:00:00Z",
      "created_by": "user_xyz789",
      "last_used_at": "2024-03-01T12:34:56Z",
      "revoked_at": null
    },
    {
      "id": "key_def456",
      "name": "Web App Key",
      "key_prefix": "cpk_def4...",
      "environment": "production",
      "key_type": "publishable",
      "allowed_domains": ["example.com", "*.example.com"],
      "created_at": "2024-01-15T00:00:00Z",
      "last_used_at": "2024-03-01T14:23:11Z",
      "revoked_at": null
    }
  ]
}
```

## Create API Key

```
POST /api/projects/{projectId}/api-keys
```

Create a new API key.

### Request Body

<ParamField body="name" type="string" required>
  Descriptive name for the API key
</ParamField>

<ParamField body="environment" type="string">
  Environment: `production` or `test`
  
  Default: `production`
</ParamField>

<ParamField body="key_type" type="string">
  Key type: `secret` or `publishable`
  
  Default: `secret`
</ParamField>

<ParamField body="allowed_domains" type="array">
  Array of allowed domains (required for publishable keys)
  
  Supports wildcards: `*.example.com`
</ParamField>

### Response

```json
{
  "apiKey": {
    "id": "key_abc123",
    "name": "My API Key",
    "key_prefix": "csk_abc1...",
    "environment": "production",
    "key_type": "secret",
    "created_at": "2024-03-01T12:00:00Z",
    "full_key": "csk_abc123def456ghi789jkl"
  }
}
```

<Warning>
The `full_key` is only returned once during creation. Store it securely - it cannot be retrieved again.
</Warning>

## Revoke API Key

```
DELETE /api/projects/{projectId}/api-keys/{keyId}
```

Revoke an API key. Revoked keys cannot be used or reactivated.

### Response

```json
{
  "success": true,
  "message": "API key revoked successfully"
}
```

## Examples

### Create Secret Key

```bash
curl https://api.cencori.com/api/projects/proj_123/api-keys \
  -H "Cookie: session=..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Backend Service Key",
    "environment": "production",
    "key_type": "secret"
  }'
```

```json
{
  "apiKey": {
    "id": "key_abc123",
    "name": "Backend Service Key",
    "key_prefix": "csk_abc1...",
    "full_key": "csk_abc123def456ghi789jkl"
  }
}
```

### Create Publishable Key

```bash
curl https://api.cencori.com/api/projects/proj_123/api-keys \
  -H "Cookie: session=..." \

    "key_type": "publishable",
    "allowed_domains": [
      "myapp.com",
```

### Create Test Key

```bash
curl https://api.cencori.com/api/projects/proj_123/api-keys \
```

```bash

```

### Using TypeScript

```typescript
const projectId = 'proj_abc123';

// Create key
async function createApiKey(name: string, type: 'secret' | 'publishable') {
  const response = await fetch(
    `https://api.cencori.com/api/projects/${projectId}/api-keys`,
    {
      method: 'POST',
      headers: {
        'Cookie': `session=${sessionToken}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        name,
        key_type: type,
        environment: 'production'
      })
    }
  );
  
  const data = await response.json();
  return data.apiKey;
}

// Usage
const newKey = await createApiKey('My Service', 'secret');
console.log('Created key:', newKey.full_key);
// Store securely!
```

## Key Types Comparison

| Feature | Secret Key | Publishable Key |
|---------|------------|----------------|
| **Prefix** | `csk_` | `cpk_` |
| **Use case** | Server-side | Client-side |
| **Domain restriction** | No | Yes (required) |
| **Exposure risk** | High | Low |
| **Should be hidden** | Yes | No |
| **Example** | Backend API | Web/mobile app |

## Key Prefixes

| Prefix | Type | Environment |
|--------|------|-------------|
| `csk_` | Secret | Production |
| `csk_test_` | Secret | Test |
| `cpk_` | Publishable | Production |
| `cpk_test_` | Publishable | Test |
| `cake_` | Agent | Special |

## Domain Patterns

For publishable keys, allowed domains support:

### Exact Match
```json
"allowed_domains": ["example.com"]
```
Matches: `example.com`

Does not match: `www.example.com`, `api.example.com`

### Wildcard Subdomain
```json
"allowed_domains": ["*.example.com"]
```
Matches: `www.example.com`, `api.example.com`, `app.example.com`

Does not match: `example.com`, `example.org`

### Localhost
```json
"allowed_domains": ["localhost"]
```
Matches: `localhost` (any port)

### Multiple Domains
```json
"allowed_domains": [
  "example.com",
  "*.example.com",
  "example.org",
  "localhost"
]
```

## Best Practices

### 1. Key Naming

Use descriptive names:

```
Good:
- "Production Backend API"
- "iOS App v2.0"
- "CI/CD Pipeline"
- "Development Testing"

Bad:
- "Key 1"
- "Test"
- "Temp"
```

### 2. Key Rotation

Rotate keys periodically:

```typescript
// 1. Create new key
const newKey = await createApiKey('Backend API v2', 'secret');

// 2. Update application
process.env.CENCORI_API_KEY = newKey.full_key;

// 3. Verify new key works
await testApiKey(newKey.full_key);

// 4. Revoke old key
await revokeApiKey(oldKeyId);
```

### 3. Environment Separation

Use separate keys for each environment:

```typescript
const config = {
  production: 'csk_prod_...',
  staging: 'csk_test_staging_...',
  development: 'csk_test_dev_...'
};

const apiKey = config[process.env.NODE_ENV];
```

### 4. Least Privilege

Create keys with minimal scope:

```typescript
// Backend: Use secret key
const backendKey = await createApiKey('Backend', 'secret');

// Frontend: Use publishable key with domain restrictions
const frontendKey = await createApiKey('Frontend', 'publishable', {
  allowed_domains: ['app.example.com']
});
```

### 5. Monitor Usage

Track key usage:

```typescript
const keys = await listApiKeys(projectId);

for (const key of keys.apiKeys) {
  const daysSinceUse = getDaysSince(key.last_used_at);
  
  if (daysSinceUse > 90) {
    console.warn(`Key "${key.name}" unused for ${daysSinceUse} days`);
    // Consider revoking
  }
}
```

### 6. Secure Storage

Never commit keys to version control:

```bash
# .env
CENCORI_API_KEY=csk_...

# .gitignore
.env
.env.local
*.key
```

Use secure secret management:

```typescript
// AWS Secrets Manager
const secret = await secretsManager.getSecretValue({
  SecretId: 'cencori-api-key'
});

const apiKey = JSON.parse(secret.SecretString).CENCORI_API_KEY;
```

## Security Considerations

### Secret Keys

<Warning>
- Never expose in client-side code
- Never commit to repositories
- Never share via email/chat
- Rotate regularly (every 90 days)
- Revoke immediately if compromised
</Warning>

### Publishable Keys

<Note>
- Safe to expose in frontend code
- Must configure allowed domains
- Monitor for unauthorized usage
- Rotate after domain changes
</Note>

## Monitoring Key Activity

Track key usage in logs:

```typescript
const logs = await fetch(
  `https://api.cencori.com/api/projects/${projectId}/logs?api_key_id=${keyId}`
);

const data = await logs.json();
console.log(`Key used ${data.requests.length} times`);
```

## Error Responses

### Invalid Key Type

```json
{
  "error": "Invalid key_type. Must be 'secret' or 'publishable'"
}
```

HTTP Status: `400`

### Missing Domains

```json
{
  "error": "Publishable keys require at least one allowed domain"
}
```

HTTP Status: `400`

### Unauthorized

```json
{
  "error": "Only organization owners can create API keys"
}
```

HTTP Status: `403`

## Troubleshooting

### Key Not Working

1. Verify key is not revoked
2. Check environment (production vs test)
3. Verify domain (for publishable keys)
4. Check project has credits
5. Review rate limits

### Domain Validation Failing

```typescript
// Debug domain validation
const origin = req.headers.get('origin');
console.log('Request origin:', origin);

const key = await getApiKey(keyId);
console.log('Allowed domains:', key.allowed_domains);

// Check if origin matches
const isAllowed = validateDomain(origin, key.allowed_domains);
console.log('Allowed:', isAllowed);
```
